Claude Code / quality loop / package-based

Claude stops when it thinks it's done. Looper keeps it going until it's actually done.

It's a native plugin that runs your quality checks - build, lint, tests - every time Claude says "finished," and pushes it back with the failures if anything is red. The code you get compiles, passes lint, and has green tests, because Claude kept iterating until those things were true. No external wrappers, no log scraping - it runs inside Claude Code's own hook system.

Why it exists

Stop acting as Claude's human quality gate.

The usual loop is wasteful: Claude writes code, you run the checks, you paste the failure, Claude fixes one thing and breaks another. Looper turns that into an internal loop instead of a handoff.

When Claude tries to stop, Looper runs typecheck, lint, test, coverage, or whatever your project demands. If a gate fails, Claude sees the failure output immediately and keeps going.

01 / Edit

Track every touched file.

Looper watches Claude's edits and runs fast local checks after each write, so problems surface while the change is still warm.

02 / Evaluate

Run your actual gates.

Typecheck, lint, test, coverage. Every gate is just a shell command, so the loop matches the real repository instead of a demo policy.

03 / Continue

Push Claude back with specifics.

Failure output is injected into the session. Claude gets another pass with concrete errors, not vague encouragement.

Auto-configuration

Detect the stack. Start from the right preset.

On first session start, Looper checks marker files and writes a matching .claude/looper.json. You get a working loop before you ever open a config file.

Claude context

What the model actually sees when the loop bites.

Looper does not say “something failed.” It sends back the real failing output, the current score, and the remaining budget.

Package system

Quality gates are only the opening move.

The kernel is intentionally small. Packages define the behavior, so the same loop mechanics can drive TDD cycles, security audits, or documentation checks.

  • session-start.shinject project context
  • pre-tool-use.shfilter or annotate edits
  • post-tool-use.shrun fast file checks
  • stop.shvote done or continue
Under the hood

A calm kernel with real constraints.

Looper is built around four Claude Code hooks, shared state, and hard stop conditions that prevent infinite churn. It stays simple enough to debug because the moving parts are plain shell and jq.

SessionStart init state, inject gate list PreToolUse track touched files, enforce budget PostToolUse run per-file checks after edits Stop evaluate gates, score, continue or finish Circuit breakers: - re-entry guard - iteration budget - two-phase stop model

No build step. No npm dependency chain. Just a loop you can inspect end to end.
